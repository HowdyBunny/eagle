# Eagle

Eagle是一个为猎头定制人才搜寻的Agentic AI系统，帮助猎头去搜寻候选人、记录文档等内容。

## 背景

很多猎头有痛点是在一些很前沿的领域，搜寻候选人是很麻烦的，并且要查看此候选人是否符合客户（即需求人才的一方）的需求和要求。同时，做出判断是非常主观且具有挑战性的

## Eagle Agent 模块

Eagle系统将会有这些Agent来帮助猎头成单。每个Agent通过现代化的方式去沟通，比如使用JSON的方式

### 1. Coordinator Agent

通过分析猎头的需求，进行调用不同的Agent进行协调工作，主要进行Router路由的工作内容，主Agent需要分析不同的Agent的工作结果，来分析下一步需要做的事情。

Coordinator Agent会对猎头的需求进行分析，负责理解意图、拆解任务、调用工具。这个Agent要通过多次向猎头提问的方式去明确需求的程度，因为猎头的话语往往比较隐性模糊，比如：客户说"我要一个懂大模型部署的技术VP，最好在新加坡，有创业经验"。这句话里"懂大模型部署"到什么程度？是做过推理优化还是训一个模型就行？"创业经验"是自己创业还是在startup待过？"技术VP"是管50人还是管5人？

同时，控制好收敛的速度：不需要多次反复提问，硬性条件第一轮问清：比如工资范围、地点等。而软性条件可以找到候选人之后再继续提问，通过猎头的反馈（这个人不错，这个人不行）进行反向精炼画像。

Agent的设计方向可以是："精准模式"——系统主导提问，3-5个问题构建完整画像。"探索模式"——猎头丢一个粗略需求，系统先出一批初筛结果，然后通过猎头的"点赞/淘汰"行为来学习偏好。两种模式可以随时切换。

CA具体的结构安排：

CA分为两层：上层是对话层，LLM处理和猎头的交互，意图识别、需求澄清。对话层负责输出一个结构化的意图JSON数据

下层是编排层，使用状态机或者是workflow engine处理任务调度、数据流传，存储的操作。下层进行执行。



### 2. Research Agent

面对比较陌生的行业，里面的一些专业名词有特定的解释。此Agent能够通过网页搜索的方式去总结行业中的一些痛点、市场分析、专业名词、技术栈，然后最终产出一篇行业的专业名词报告。学习到的行业黑话和专业名词要能够去优化搜索策略

负责动态学习陌生行业，丰富系统的内部知识图谱。

在后续项目阶段，如果有合适的情报追踪接口，这个Agent或许还能够去追踪行业的职位变动、融资新闻、业务调整等等。

疑惑的点：数据流应该如何handle？这些信息应该变成什么？是一个"LLM推理工程师的技能标签树"？还是一段自由文本存进向量库？

### 3. Evaluator Agent ：

负责简历深度解析与岗位人选的匹配度判断。Agent 会结合“客户的原始 JD（职位描述）”、“猎头的补充要求”以及“隐性软技能需求”，对候选人进行多维度的打分。不仅输出“匹配/不匹配”，还要生成一段**“推荐理由（Why this candidate）”**和**“风险提示（面试需深挖的点）”**

匹配度打分的评分框架：硬性条件匹配度（技能、年限、地点、薪资）占多少权重，软性条件匹配度（行业背景、管理经验、文化适配）占多少权重，这些权重应该可以被猎头调整。

对于推荐理由：猎头可能需要把推荐理由发给客户，所以输出的语言风格要专业、简练、有说服力。这需要针对性的prompt工程，不是通用的"总结匹配点"能做好的。

后续的开发方向：反馈闭环：如果猎头觉得打分系统不准，Agent应当需要进行学习。所以纠正机制也需要进行设计



## 如何去获取人才：

Linkedin和猎聘是非常反爬的，消耗非常大，账号非常容易被封。对于模拟Computer Use的方式去处理信息，Token消耗会非常非常大，而且经常需要猎头去交互以识别反爬机制。

所以，使用Chrome的插件，当猎头进行人才浏览的时候，Agent 通过插件直接读取经过清洗之后的网页 DOM 结构（插件需要有解析脚本），提取结构化数据。猎头有控制权，在网页的右边有一个一键收录的功能按钮。

此Agent会将候选人的特定的信息的进行存储（剔除无关信息）。

**混合检索（Hybrid Search）架构**。

- 人才入库时，必须将数据拆分：**标量数据**（地点、职级、薪资、入库时间）存入关系型数据库（如 PostgreSQL）；**文本数据**（工作经历描述、专业技能总结）向量化后存入 Vector DB（如 Milvus 或 Pinecone）。

风险：DOM结构会有变化，需要持续维护DOM解析规则。

采集了人才之后直接触发Evaluator Agent进行匹配打分。

DOM维护方面，考虑用大模型做fallback——当规则提取失败时，把页面的文本内容丢给模型做非结构化提取。

## 人才池管理：

RAG搜索：引入向量数据库，Research Agent搜索到的内容不仅是报告，是要存入到系统的行业知识库。当进行人才的需求匹配的时候，需要查找的是语义级别的相似度

信息过时的处理方式：将人才信息的入库时间一并计入到SQL数据库中。

如何判断一个人是否在看机会：在存入库的时候：判断此人是否在看机会：比如Linkedin上的头像是否是Open to work，或者最近的动态是否更新频繁等等方式

信息更新的机制：给每个候选人一个"信息置信度"评分，综合考虑入库时间、**任期周期（Tenure）**：在同一岗位待满 3-4 年。当置信度低于阈值时，系统在返回搜索结果时标注"信息可能过时，建议重新查看此人LinkedIn"。

一些问题明确：

1. 不考虑同个公司不同Hunter是否会有候选人冲突，因为不同的Hunter实际负责的区域不太一样，冲突可能性低。

2. 隐私合规性：需要进行披露并进行审计，但本项目非公开面对市场，而是给个人使用

3. hunter的反馈闭环：猎头往往不会主动去调节默认的评分权重，所以当猎头进行反馈的时候，比如说“这个人技术够了但管理经验太少”，CA就要开始将权重进行调节，追加到便好记录作为上下文，EA要读取相关的调节后的数据进行评分

4. “信息置信度”的简单机制：**基准分**：100分。

   **时间衰减**：入库每超过 1 个月，扣 5 分。

   **任期预警**：当前工作时长处于 30-40 个月（快满 3 年/期权兑现期），**加 20 分**（代表异动可能性极高）。当前工作时长小于 1 年，扣 30 分（刚跳槽，除非被裁，否则极难挖）。

   最终分数低于 60 分，系统就打上 `[⚠️ 信息可能过时 / 需激活]` 标签。
