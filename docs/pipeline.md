**冷启动阶段**：

对于一个agenttic系统，空人才池，空项目。Coordinator Agent为CA，Research Agent为RA，Evaluator Agent为EA

hunter先对CA说，需要新建一个项目，客户是xxx，招聘的人的需求是xxx。

CA会在指定的项目目录创建一个文件夹，并且进行特定的命名（可以通过小型LLM进行操作或者直接操作即可，优先后者）。然后CA会通过LLM进行需求分析，CA知晓此系统有什么其他的Agent工具。LLM通过判断hunter的描述，判断语句是否隐晦，然后进行反问，如果需求文档中没有明确一些硬性条件（地区、学历、经验），需要在第一轮进行问清楚。如果hunter要求CA先进行流程，则回答：好的，等匹配到一些候选人之后再进行继续细化要求。

在这个过程中CA会通过回答和hunter提交的文件进行候选人画像构建，然后存储到项目文件目录到文档中，记录好沟通细节和时间等内容。



当hunter需要去了解行业的内容的时候，可以通过CA说：总结一篇关于储能行业的猎头学习文件。

CA会调动RA去开始搜索。RA需要进行网页搜索，市场分析、专业名词、技术栈、行业黑话，然后总结一篇文档存储在项目子文件夹中。这些信息会通过embedding的方式对每个内容进行向量化，从而存储在一个向量数据库中。

RA的输出需要参考PRD文档。总共有可以供hunter去阅读的、存入到向量数据库的、结构化输出的



hunter在浏览LinkedIn 猎聘的时候，Chrome插件在右边，当hunter进行点击收录的时候，插件会将网页的DOM进行清洗（纯前端），形成结构化数据，然后直接触发Agent工作流，然后由CA（编排层）进行整理结构化的数据，将数据纳入到人才池中（需要对数据进行拆分为标量数据存入SQL和文本数据存入向量化数据库中）（Note：如果DOM太复杂，需要小型的LLM进行保底机制去处理）。CA然后触发EA进行人才的匹配度评分。EA会使用新的候选人的信息+项目的中需求信息进行+RA搜索到的行业知识库+Hunter预先设置的评分框架（不同条件的匹配度权重）进行匹配评分。不仅生成匹配的百分比，还要生成一段**“推荐理由（Why this candidate）”**和**“风险提示（面试需深挖的点）”**

当EA的任务结束的时候，将结果返回给CA，CA再与hunter进行交互沟通



**当有丰富的人才池阶段**

hunter会对CA进行表述新建一个项目，过程和冷启动阶段一样。

当hunter对CA说：向人才池中去找有没有比较匹配的人才，CA会直接向人才池中提取人才，然后提交给EA进行评估和匹配度分析。

CA可以进一步细化，比如通过混合检索的方式：去先使用SQL硬性指标去筛选部分的人，然后可以通过Vector DB进行语义粗筛选（保留一定的人数，防止token爆炸），最后再提交给EA进行深度语义检索并返回结果评分和理由

同时CA会检查人才库的记录，进行信息置信度评分。判断此人才是否需要进行信息更新（提供hunter其原来的猎聘、Linkedin网址）。



**工作流的异常路径处理**

1. 猎头中途更改需求：客户说不需要只在新加坡，全球都可以，那么CA重新处理画像，EA要进行重新打分，被SQL过滤的候选人要重新被检索出来。这可以被设计为：混合检索不能是"先SQL过滤再向量搜索"的串行流程，而应该是两路并行，最后合并排序
2. 猎头多个项目一起工作：需要切换文件夹，但是猎头收录的候选人进入了全局的人才池。但是触发的EA应该是需要在当前项目中进行匹配评分。可能的解决方案：在插件的收录的时候：默认是存入人才池并匹配现在正在处理的项目。猎头也可以切换为：存入总库。当猎头选择存入总人才池的时候，不需要CA触发EA
